The Tea Stall Data Scientist
Howrah · Every Morning · Unaware Data Science

Ravi has run a tea stall at Howrah station for 22 years. He has never touched a computer. But every morning, his brain is running one of the most sophisticated data pipelines on the planet.

Before any algorithm touches your data, it must go through preprocessing — a series of transformation steps that convert raw, messy, real-world data into a form that a machine learning model can actually learn from.

🌾 The Wheat-to-Bread Analogy

Raw data is like wheat harvested from a field. The wheat is real and valuable — but you cannot eat it directly. You must mill it into flour (clean), mix it with water and yeast (transform), knead it (normalise), and bake it (encode) before it becomes bread that humans can consume.

Data preprocessing is exactly this. The AI model is the person eating the bread. It cannot consume raw wheat (raw data). Each preprocessing step transforms the raw material into something the model can digest and learn from.

Preprocessing in Python — What Each Step Looks Like

# The preprocessing pipeline — what you'll actually write in your Masters programme # Each line below corresponds to one preprocessing step import pandas as pd # STEP 1: Load data df = pd.read_csv('customer_data.csv') print(df.shape) # → (85000, 18) — 85,000 rows, 18 columns # STEP 2: First look — always do this before anything else df.head(5) # First 5 rows df.info() # Data types + null counts df.describe() # Stats: min, max, mean, std for numeric columns df.isnull().sum() # Count missing values per column # STEP 3: Fix types df['price'] = df['price'].str.replace('₹','').str.replace(',','').astype('float') df['order_date'] = pd.to_datetime(df['order_date'], dayfirst=True) # STEP 4: Handle missing values df['age'].fillna(df['age'].median(), inplace=True) # STEP 5: Remove duplicates df.drop_duplicates(inplace=True) # STEP 6: Encode categorical column for ML df['gender_encoded'] = df['gender'].map({'Male': 0, 'Female': 1, 'Other': 2})
💡 The Golden Rule of Preprocessing

Always perform preprocessing steps in the correct ORDER. You cannot normalise data before fixing data types (you'd be normalising strings). You cannot encode categories before handling missing values (the NaN would become its own category). The sequence matters as much as each individual step.

Preprocessing Pipelines in Production — Scikit-learn's Pipeline Object

In industry, preprocessing steps must be repeatable, testable, and automatically applied to new data. This is done using Pipeline objects that chain all preprocessing steps together.

# Industry-grade preprocessing pipeline # This exact pattern is used in companies like Razorpay, PhonePe, Paytm from sklearn.pipeline import Pipeline from sklearn.preprocessing import StandardScaler, OneHotEncoder from sklearn.impute import SimpleImputer from sklearn.compose import ColumnTransformer # Define which columns are numeric vs categorical numeric_cols = ['age', 'income', 'credit_score', 'loan_amount'] categorical_cols = ['city_tier', 'employment_type', 'product_category'] # Build separate pipelines for each type numeric_pipeline = Pipeline([ ('impute', SimpleImputer(strategy='median')), # fill NaN with median ('scale', StandardScaler()), # Z-score normalisation ]) categorical_pipeline = Pipeline([ ('impute', SimpleImputer(strategy='most_frequent')), # fill NaN with mode ('encode', OneHotEncoder(handle_unknown='ignore')), # text → numbers ]) # Combine both into one transformer preprocessor = ColumnTransformer([ ('num', numeric_pipeline, numeric_cols), ('cat', categorical_pipeline, categorical_cols), ]) # Apply the same preprocessing to BOTH training data AND new live data # This prevents "training-serving skew" — a major source of production failures X_train_processed = preprocessor.fit_transform(X_train) X_new_processed = preprocessor.transform(X_new) # Uses same rules, no refitting
⚡ Why Pipelines Prevent a Common Industry Disaster

A major Indian fintech company built a fraud detection model. During training, they manually computed the mean income as ₹4.2L and used it to fill nulls. The model performed brilliantly.

Six months later, in production, the live scoring code was filling income nulls with ₹6.8L (the current mean, which had shifted with the customer base). The training distribution and production distribution were different — the model was silently underperforming for 6 months before anyone noticed the fraud rate creeping up.

This is called training-serving skew. Scikit-learn Pipelines prevent it by storing the preprocessing parameters (mean, std, categories) from training time and reusing them exactly at prediction time. The preprocessing is locked in, not recomputed.

Public and Private Data

Not all data is created equal — and not all data is yours to use freely. Understanding the difference between public and private data is both a technical and an ethical cornerstone of data science.

1Foundation
2Practical
3Industry
📚 The Library vs Private Diary Analogy

A public library has books anyone can walk in and read. No permission needed. No cost. The books are there for public benefit — government reports, academic papers, historical records. This is public data.

Your personal diary is private. Someone reading it without your permission is a violation — legally and ethically. Even if they could physically access it, doing so is wrong. This is private data.

In data science, this distinction matters enormously. Using private data without authorisation is illegal (IT Act, PDPB in India), unethical, and has ended careers. Knowing which category your data falls into is the first question to ask — before writing a single line of code.

The 4 Categories of Data by Access Rights

🌐
Open Public Data

Free for anyone. No restrictions. Government releases, academic datasets, open-source repos. Examples: Census India, RBI data, WHO statistics, Kaggle datasets, Wikipedia.

🔑
Licensed Public Data

Publicly available but with usage restrictions. Must agree to terms. Examples: Google Maps API (free up to a limit), commercial weather APIs, Bloomberg financial data.

🏢
Private Organisational Data

Owned by a company. Employees can use it for work purposes. Sharing outside is prohibited. Examples: Flipkart's transaction database, Jio's subscriber records, HDFC's loan data.

🔒
Personal Private Data

Belongs to individual people. Highest legal protection. Cannot be used without informed consent. Examples: Aadhaar, medical records, financial statements, biometric data, personal messages.

⚠️ The "Publicly Available" Trap

A common mistake: assuming that because data is technically accessible online, it can be freely used for any purpose. This is wrong in several important ways:

  • Website Terms of Service: LinkedIn prohibits scraping user data, even though profiles are publicly visible. HiQ Labs scraped LinkedIn data, won a court case, but the legal battle cost millions and changed policy.
  • Aggregation attacks: Individual public data points combined create private profiles. Your name (public) + your employer (public) + your neighbourhood (public) + your health condition (public from a news article) = a detailed private profile you never consented to.
  • Purpose limitation: Under DPDPA, data collected for one purpose (voting) cannot be used for another (commercial targeting) even if technically accessible.

In your Masters programme, always check: (1) Do I have the legal right to use this data? (2) Was it collected with informed consent for this use? (3) Does using it respect the people it describes?

Loading Excel Files — The Hidden Complications

📊 Problem: Multiple Sheets in One Excel File ▼

An Excel file from the Finance team has 4 sheets: "Q1_Sales", "Q2_Sales", "Q3_Sales", "Q4_Sales". You need to combine all 4.

# Load all sheets and combine them all_sheets = pd.read_excel('annual_sales.xlsx', sheet_name=None) # None = load all combined = pd.concat(all_sheets.values(), ignore_index=True) print(combined.shape) # Now all 4 quarters in one dataframe
📊 Problem: Excel Has Merged Cells and Formatting ▼

Business Excel files often have merged header cells, colour-coded cells, and rows that are summaries, not data. These cause loading errors or incorrect column assignments.

Best practice: When receiving Excel files for analysis, ask the sender to "Save as CSV (comma delimited)" first. This strips all formatting and gives you clean tabular data. Alternatively, use the skiprows and header parameters to navigate past formatted sections.

📦 Industry: How Meesho Handles 500GB Daily Data Exports

Meesho generates approximately 500 GB of raw transaction and event data daily. Loading this as CSV into pandas would require a server with 2+ TB RAM — impractical and expensive. Their solution uses a multi-format strategy:

  • Raw clickstream events → Parquet in S3 (compressed from 500 GB to 42 GB)
  • Daily transaction summaries → Parquet partitioned by date and category
  • Real-time fraud scores → JSON via Kafka streams

  • Business reports → Exported to Excel only at the final summary layer

The format choice at each layer reduces total infrastructure cost by ~70% compared to storing everything as CSV. A data scientist joining Meesho's team must understand Parquet, Kafka, and HDF5 on Day 1 — not just CSV and Excel.

HDFC Bank — The ₹800 Crore Cost of Dirty Data
Banking · Credit Risk · 2019 · Real Incident

In 2019, a major Indian bank conducted a post-mortem on ₹800 crore in written-off bad loans. The primary finding was unexpected: the credit scoring AI model was performing correctly. The problem was the data it was trained and scored on.

Three specific data quality failures were identified:

Monthly vs Annual Income Mix-Up

12% of "Annual Income" fields contained monthly salary figures. A customer earning ₹60,000/month had "60000" entered where the model expected annual income of "720000". The model saw this as an extremely low-income person and either rejected legitimate loans (losing business) or, in some configurations, classified them as a different risk segment entirely.

Decimal Point Errors

8% of entries had decimal errors: ₹1,20,000 entered as ₹12,000. These appeared as very low income customers. The model approved smaller loan amounts than the customer actually qualified for — either frustrating customers or exposing the bank to over-leverage risk.

For self-employed applicants, loan officers who couldn't verify informal income entered "0" instead of leaving the field blank (NULL). The model treated 0 as "no income" rather than "income unknown" — a critical distinction. These applicants were systematically disadvantaged.

💡 None of these were AI failures. The model did exactly what it was built to do. They were data quality failures at the point of entry, propagated through to model scoring. The fix required retraining the model with cleaned data AND implementing real-time validation at the data entry screen.

The Data Cleaning Workflow in Professional Practice

Profiling (Day 1)

Automated statistical summary of every column: data type, % null, min/max/mean, unique value count, most common values. This gives you a map of where problems are hiding. Tools: pandas-profiling, Great Expectations, Ataccama.

Write the Data Contract (Day 1-2)

Before touching any data, document what "good data" looks like. Write validation rules for every column. This is your reference standard for all cleaning decisions. This step prevents disputes later about whether something is a bug or a feature.

Structural Fixes (Day 2-3)

Fix data types, drop irrelevant columns, rename columns to a consistent naming convention, handle structural issues like merged header rows, footer totals.

Deduplication (Day 3)

Remove exact duplicates first. Then investigate near-duplicates using fuzzy matching. Escalate ambiguous cases for domain expert review rather than making assumptions.

Missing Value Treatment (Day 4-5)

Analyse patterns: is missingness random or systematic? Decide treatment per column based on business impact, % missing, and downstream use. Document every imputation decision.

Invalid Value Detection and Repair (Day 5-6)

Run validation rules from the data contract. Flag violations. Fix what can be fixed programmatically. Escalate what requires human judgment or source system investigation.

Outlier Analysis (Day 6-7)

Statistical detection (IQR, Z-score) plus visual exploration. Distinguish data errors from legitimate extremes. Domain expert involvement is essential for this step.

Standardisation (Day 7-8)

Consistent text formats, date formats, unit conversion, encoding of categorical variables.

Final Validation + Documentation (Day 9)

Re-run the data contract validation suite. All rules should pass. Produce a Data Quality Report: what was found, what was changed, what was escalated, what known issues remain.

🏆 Pro Practice: Always Write a Data Cleaning Report

This document protects you legally (GDPR/data audits), enables reproducibility (anyone can re-run your process), and is required for regulated industries (pharma, banking, healthcare). It also enables model debugging: if a model predicts strangely 6 months later, you can trace it back to data cleaning decisions made during preparation.

Data Types

Every value in a dataset has a type. Getting it wrong is like trying to add apples and oranges — the computer either crashes, silently gives wrong answers, or wastes enormous memory.

1Foundation
2Practical
3Industry
📦 The Container Analogy

Imagine you have different containers in your kitchen: a water bottle, a sugar jar, an egg carton, a spice rack. Each is designed for one type of thing. If you pour water into the sugar jar, you ruin the sugar. If you try to put an egg in the water bottle, it doesn't fit.

Data types are exactly the same principle. An Integer container holds only whole numbers. A String container holds text. If you pour text into an Integer container, the computer either rejects it or silently corrupts it — just like water in a sugar jar ruins what was already there.

The 7 Core Data Types — With Stories

🔢 Integer — Whole Numbers Only ▼

What it is: Whole numbers with no decimal part. Can be positive or negative.

Real examples: Number of students in class (42, not 42.5), floors in a building, number of orders, age in years, PIN code.

Critical trap — PIN codes and IDs: Delhi's PIN code is 110001. If stored as Integer, it stays 110001. But Goa's PIN 403001 also fine. HOWEVER: if you ever need to pad with zeros or compare as text, Integer breaks things. This is why you should often store codes as strings even when they look like numbers.

Memory fact: An integer takes 4-8 bytes. A text/string of the same number takes up to 10+ bytes. For a column with 100 million rows, choosing Integer over String for numbers can save gigabytes of memory.

🔣 Float/Decimal — Numbers with Precision ▼

What it is: Numbers that can have a fractional part (after the decimal point).

Real examples: Price ₹99.99, temperature 36.7°C, GPS latitude 18.9220° N, interest rate 8.5%, weight 68.3 kg, Ola surge multiplier 1.8x.

The ₹1 vs ₹0.99 problem: If you store price as Integer, ₹99.99 becomes ₹99. For 10 million transactions, you've undercharged by ₹10 million! Flipkart's entire pricing system depends on decimal precision.

Float precision warning: Computers represent decimals in binary, which causes tiny rounding errors. 0.1 + 0.2 in a computer is actually 0.30000000000000004, not 0.3. For money calculations, banks use a special "Decimal" type that avoids this issue. This is why your bank balance never shows a fraction of a paisa.

📝 String/Text — Any Characters ▼

What it is: Any sequence of characters: letters, numbers, symbols, spaces. The most flexible type.

Real examples: Names ("Vineet Jhabak"), Aadhaar number ("1234 5678 9012"), PAN ("ABCDE1234F"), addresses, Zomato reviews, product descriptions.

The "9" + "1" = "91" trap: If you store numbers as text and try to add them, Python does string concatenation instead of arithmetic. "9" + "1" = "91", not 10. This has caused real financial calculation errors in banking systems where amounts were accidentally stored as text.

Why Aadhaar must be String: Aadhaar numbers start with 1-9. Some start with lower digits. More importantly, you never CALCULATE with an Aadhaar number — you just store and look it up. Storing as Integer is wrong because (a) you might lose leading digits in edge cases, (b) doing arithmetic on ID numbers is meaningless.

📅 DateTime — Time is Special ▼

What it is: Stores a specific point in time. Has special arithmetic abilities that text doesn't have.

Real examples: Order placed: 2024-01-15 14:32:07. Flight departs: 2024-03-20 06:15:00 IST. Birthday: 1995-08-22.

Why DateTime beats text for dates: If "Order_Date" is stored as text "15-Jan-2024", you cannot ask: "How many days since this order?" The computer doesn't know "15-Jan-2024" is a date. But if stored as DateTime, you can do: Today (2024-03-10) - Order_Date (2024-01-15) = 55 days. This powers every "Expected delivery in X days" calculation on every e-commerce site.

IRCTC reservation example: IRCTC must enforce: you cannot book a train ticket for a date in the past. This validation only works if booking_date is stored as DateTime. If it's text, "01-Mar-2023" comes alphabetically AFTER "31-Dec-2022" in some sorting orders — and the validation fails.

✅ Boolean — The Yes/No World ▼

What it is: Only two possible values: True or False (also stored as 1 or 0). The most memory-efficient type (1 bit!).

Real examples: Is_Active_Customer: True/False. Has_Aadhaar: True/False. Is_Verified_Seller (Flipkart): True/False. Is_Weekend: True/False.

The inconsistency problem: Without enforced Boolean type, the same yes/no question gets answered as: "Yes", "yes", "YES", "Y", "y", "1", "True", "true", "TRUE", "X" — all meaning the same thing. When you filter for active customers with is_active = "Yes", you miss all the "true" and "1" entries. Bangalore telecoms lost 8% of their "active subscriber" count in a report because of this exact issue.

🗂️ Categorical — Fixed Set of Options ▼

What it is: A column that can only take one value from a predefined list. Like a multiple-choice question.

Real examples: Blood type (A/B/AB/O), Gender, City tier (Tier-1/2/3), Employment type (Salaried/Self-employed/Retired), Product category, Train class (1AC/2AC/3AC/SL/General).

Why it matters for memory and AI: If "Product_Category" is stored as plain text, pandas/Python treats each unique string as independent. With 50,000 products all in "Electronics", it stores "Electronics" 50,000 times. As Categorical type, it stores the label once and just numbers (0,1,2...) in each row. 10-100x memory reduction for high-cardinality repeated categories. Large datasets can become workable from memory-crushing.

Encoding for ML: Machine learning models only understand numbers. Categorical columns must be converted: "Salaried" → 0, "Self-employed" → 1, "Retired" → 2. This is called "Label Encoding". More advanced is "One-Hot Encoding": create a separate column for each category with 0/1. Essential before any ML model training.

Handling Missing Values

In every real dataset, some values will simply be absent. Your decision about how to handle each gap — fill it, delete it, leave it — can make or break your entire model.

1Foundation
2Practical
3Industry
🧩 The Jigsaw Puzzle Analogy

You are solving a 1000-piece jigsaw puzzle. You finish assembling it, but 40 pieces are missing. The 40 gaps are your missing values.

You now have exactly 3 choices — and each has consequences:

  • Option 1: Ignore the gaps. The puzzle still mostly shows the picture. This works if only a few pieces are missing in non-critical areas. Risk: the gaps might be exactly where the main subject of the picture is.
  • Option 2: Cut out the sections with gaps. You physically remove the incomplete sections of the puzzle. Now you have a smaller but complete puzzle. Risk: you've permanently lost information — and the removed sections might have contained something important.
  • Option 3: Fill the gaps with your best guess. Based on surrounding pieces, you paint in what you think was there. Risk: if you guess wrong, you've permanently added incorrect information to the puzzle.

There is no universally correct choice. The right approach depends on: how many pieces are missing (2 vs 200), where they are (border vs centre), and how important that area is to understanding the picture.

It has been blinded to a key risk signal. The missingness itself WAS the signal.

The 6 Ways to Handle Missing Values

🗑️
Delete the Row

Remove rows where critical values are missing. Safe when <2% are missing and missingness is random.

✂️
Delete the Column

Drop the entire column when >60% is missing. A column that's mostly empty adds noise, not value.

⚖️
Fill with Mean

Replace with column average. Only for symmetric numerical distributions without outliers.

📊
Fill with Median

Replace with the middle value. Better than mean when outliers distort the average.

📌
Fill with Mode

Replace with most common value. For categorical columns or when you need a default text value.

🏷️

Mean vs Median — When It Really Matters

💰 The Billionaire in the Room Problem

A classroom of 30 students has monthly income data (from part-time jobs). 28 students earn between ₹3,000 and ₹8,000/month. One student is a startup founder earning ₹1,50,000/month. One more earns ₹5,000/month.

Mean (average): Total all incomes, divide by 30. The startup founder's ₹1,50,000 pulls the average up significantly. Mean = ~₹10,300. This is higher than 29 out of 30 students actually earn. If you fill missing income values with this mean, you're overstating typical income.

Median (middle value): Sort all 30 by income, take the middle one. = ~₹5,500. This represents what a typical student actually earns, regardless of the outlier.

Rule of thumb: If the data has outliers, use median. If it's symmetric with no extreme values, mean and median will be similar — either is fine.

Handling Outliers

An outlier is a value so far from the rest that it stands alone. The critical question is never "should I remove it?" — it's always "WHY is it extreme?" That answer changes everything.

1Foundation
2Practical
3Industry
🏏 The Cricket Match Analogy

Your college cricket team plays 10 matches this season. Team scores: 182, 167, 194, 171, 188, 176, 195, 169, 178, 14.

That 14 is our outlier. It is wildly different from every other score. But notice: there are TWO completely different explanations:

  • Explanation A — Data Error: A student was typing in scores and accidentally typed "14" instead of "174". This is a transcription error. The correct action is to fix it to 174 (if you can verify from the original scorecard).
  • Explanation B — Real Event: That match was played in a cyclone warning, 6 players were sick with fever, the opposing team had a professional-level fast bowler on loan, and the match was abandoned after 5 overs. That 14 is REAL and important. Removing it would misrepresent the team's actual performance record.

The outlier is identical in both cases — it's your investigation and context that determines whether to fix it, remove it, or keep it with a note. This is why outlier handling requires human judgment, not just algorithms.

The 4 Things an Outlier Can Be

✏️
Data Entry Error

Someone typed 14 instead of 140. Age is -5. Score is 110%. These should be corrected or removed.

⚙️
Measurement Error

Sensor malfunctioned. Scale wasn't calibrated. Default value (-999) was never replaced. These are artefacts, not reality.

🌟
Real but Rare

A genuine extreme event: Sachin Tendulkar's batting average in a dataset of cricketers. The billionaire in an income dataset. Real, but extreme.

🔍
The Signal (Most Valuable)

A fraudulent transaction. A medical emergency. A cyber-attack. The outlier IS the thing you're looking for. Removing it would be catastrophic.

  • Temperature = -9999 (sensor default error)
  • Score = 110% (maximum is 100)
  • Salary = ₹0 for a full-time employee
  • Transaction amount = ₹0.00001
✓ Outliers to Keep (Real Signals)

Method 1: The IQR Method — Step by Step

📊 Worked Example: Finding Salary Outliers

You have annual salaries (₹ Lakhs) for 100 employees. After sorting, you find:

Step 1: Find Q1 (25th percentile) = ₹3.8L — value below which 25% of salaries fall

Step 2: Find Q3 (75th percentile) = ₹9.2L — value below which 75% of salaries fall

Step 3: Calculate IQR = Q3 - Q1 = 9.2 - 3.8 = 5.4

Step 4: Calculate fences:

Lower Fence = Q1 - (1.5 × IQR) = 3.8 - (1.5 × 5.4) = 3.8 - 8.1 = -4.3L Upper Fence = Q3 + (1.5 × IQR) = 9.2 + (1.5 × 5.4) = 9.2 + 8.1 = 17.3L

Step 5: Flag outliers: any salary below -4.3L (impossible, so no lower outliers) or above 17.3L (flags the CEO at ₹85L, the CTO at ₹42L, and 3 senior VPs at 18-25L).

Now you INVESTIGATE those 5 flagged records. Are they errors or real senior executives? In this case, all 5 are real. You keep them but segment the analysis: "Executive compensation" vs "Staff compensation" — two different distributions, one dataset.

Method 2: Z-Score — How "Abnormal" Is a Value?

Z-Score = (Value - Mean) / Standard Deviation Example: Average exam score = 72, Standard Deviation = 9 Student A scored 94: Z = (94-72)/9 = +2.44 → High but within normal range Student B scored 12: Z = (12-72)/9 = -6.67 → Extreme outlier (6.67 SDs below mean!) General rule: |Z| > 3 → investigate as potential outlier

Standardising Values

Standardisation enforces uniformity — so the same thing always looks the same, regardless of who entered it, when, or from which system. Without it, your analysis will treat "Mumbai" and "mumbai" as different cities.

1Foundation
2Practical
3Industry
📏 The NASA Unit Disaster — This Actually Happened

In 1999, NASA lost a $327 million spacecraft — the Mars Climate Orbiter — because of a standardisation failure. One team's software output thrust values in pound-force seconds. The receiving system expected Newton seconds. Nobody standardised the units.

The spacecraft entered Mars orbit at the wrong altitude and was destroyed.

This is not a rare mistake. In data science, unit mismatches, format mismatches, and naming inconsistencies cause silent errors that are far harder to detect than system crashes — because the calculation runs successfully, it just produces wrong answers.

The 7 Most Common Standardisation Problems

🔤 Problem 1: Inconsistent Text Case ▼

The problem: One column has "Mumbai", "mumbai", "MUMBAI", and "Bombay" — all referring to the same city.

What breaks: GROUP BY city in SQL gives you 4 separate groups instead of 1. "How many customers in Mumbai?" returns 1/4 of the real answer. Revenue reports show Mumbai as having multiple "branches."

The fix: Standardise to Title Case (first letter of each word capitalised). Also handle historical aliases: "Bombay" → "Mumbai", "Madras" → "Chennai", "Calcutta" → "Kolkata".

Real impact: Reliance Retail discovered their customer database had 47 distinct text variants for "Mumbai" across 3 merged datasets. A sales dashboard was reporting ₹450 crore in revenue for "Mumbai" when actual Mumbai revenue was ₹2,100 crore — because 87% of Mumbai entries were in non-standard format.

📅 Problem 2: Mixed Date Formats ▼

The problem: A date column collects from multiple systems and shows: "15/01/2024", "15-Jan-2024", "January 15, 2024", "2024-01-15", "15.1.24", "01/15/2024" (American format).

What breaks: Sorting by date gives random results. "15/01/2024" vs "01/15/2024" — which is January 15 and which is March 1 in American format? The computer can't tell without context.

The fix: Convert everything to ISO 8601 standard: YYYY-MM-DD (2024-01-15). This is unambiguous, sortable alphabetically AND chronologically, and internationally recognised.

Business story: An insurance company's claims system had date entries from 3 different vendor portals. A claim from "01/03/2024" was interpreted as January 3rd by one system and March 1st by another. A patient was denied a claim because the "treatment date" appeared to be after the "policy expiry date" — due to format confusion. The correct date was 62 days before expiry.

💰 Problem 3: Currency in Number Fields ▼

The problem: Price column has: "₹1,29,999", "Rs. 129999", "INR 130K", "1,29,999.00", "129999".

What breaks: Arithmetic fails completely. You can't add "₹1,29,999" and "129999" — one is text, one is a number. Even if both are text, the comma in "1,29,999" means you're storing "1" and "29" and "999" as three separate things.

The fix in steps: (1) Identify the currency (₹, Rs., INR → all are INR). (2) Remove the symbol. (3) Remove commas. (4) Handle "K" notation: "130K" → 130000. (5) Convert to float. "₹1,29,999" → 129999.0

📞 Problem 4: Phone Number Formats ▼

The problem (all the same Indian phone number): "9876543210", "+91-9876543210", "+91 98765 43210", "09876543210", "98765-43210", "91-9876543210"

What breaks: Deduplication fails — these all look like different records. WhatsApp/SMS campaigns send the same message to the same person 6 times (or fail to reach them because the system looks up "9876543210" but only has "+91-9876543210").

The fix: Strip all non-digits → "91 9876543210". Remove country code (91) → "9876543210". Validate: exactly 10 digits, starts with 6, 7, 8, or 9. If the result doesn't match these rules, flag as invalid rather than guessing.

Fixing Invalid Values & Filtering Data

Invalid values are present but wrong. They're more treacherous than missing values — at least with missing values, you know something's absent. Invalid values look legitimate at first glance.

1Foundation
2Practical
3Industry
🎭 The Counterfeit Note Analogy

A missing value is like an empty slot in your wallet — you know immediately something's missing.

An invalid value is like a counterfeit ₹500 note in your wallet — it's there, it looks real at a glance, but it's wrong. You might spend it without realising. The person who receives it will discover the problem later, far downstream from where the error originated.

This is why invalid values are often more dangerous than missing values. A missing value stops calculations. An invalid value corrupts them silently.

The 5 Categories of Invalid Values — With Stories

📏 Category 1: Range Violations — Values Outside the Possible Range ▼

Examples: Age = -3 or 312. Score = 108% or -15%. Price = -₹500. Completion percentage = 115%.

Why they happen: No input validation on the form. Copy-paste errors. System default values that never got replaced (-1 or -999 meaning "unknown"). Manual override without sanity checks.

How to detect them: Set business rules: Age must be between 0 and 120. Score must be between 0 and 100. Price must be greater than 0. Run every column against its rules; flag violations.

Real story: A government hospital's anesthesia dosage calculation system used patient weight as input. A data entry error put age (68 years) into the weight field (68 kg was correct, but for a different field). The system calculated the correct dose for a 68-year-old patient, not a 68 kg patient. The weight field had no minimum/maximum validation. This near-miss led to mandatory range validation on all clinical data entry systems in the state.

📧 Category 2: Format Violations — Wrong Pattern ▼

Examples: Email "priya@" (missing domain). PAN "ABCDE1234" (should end in a letter). Phone "9876543" (only 7 digits). IFSC "HDFC0000" (should be 11 chars).

How to detect them: Use regular expressions (regex) — patterns that define what valid data looks like.

# Regular expressions — rules written in pattern language Indian mobile: ^[6-9][0-9]{9}$ ← starts with 6-9, followed by exactly 9 more digits Email address: ^.+@.+\..+$ ← characters, @, characters, dot, characters PAN number: ^[A-Z]{5}[0-9]{4}[A-Z]$ ← 5 uppercase letters, 4 digits, 1 uppercase letter IFSC code: ^[A-Z]{4}0[A-Z0-9]{6}$ ← 4 letters, 0, 6 alphanumeric

Any value that doesn't match its pattern is invalid — regardless of what it contains. The system doesn't guess; it flags and requires human review or resubmission.

🔗 Category 3: Cross-Field Violations — Inconsistent with Other Fields ▼

Examples:

  • Date_of_Birth = 2010 AND Employment_Start = 2015 → 5-year-old employee
  • Retirement_Date = 2018 AND Joining_Date = 2022 → retired before joining
  • Blood Type = AB AND Child's Blood Type = O (genetically impossible for certain parent combinations)
  • Graduation_Year = 2019 AND Experience_Years = 15 (would have graduated at age 7)
  • Policy_Start = 2020-01-01 AND Claim_Date = 2019-12-15 → claim before policy started

Why they're dangerous: Each field individually looks valid. Age = 12 is valid. Employment date = 2015 is valid. Only when you look at them TOGETHER does the error emerge. These require cross-field validation rules, not just single-column checks.

Real insurance case: An insurance company had 4,200 claims where the "Date of Treatment" was before the "Policy Activation Date." The AI model trained on this data learned that pre-activation claims were normal (because they were in the training data). The model then failed to flag genuinely fraudulent retroactive claims — costing ₹12 crore before the training data was fixed.

🗂️ Category 4: Category Violations — Not in the Allowed Set ▼

Examples: Blood Type = "Green". Gender = "Alien". State = "Maharastra" (misspelling). Country = "Bharat" (not in the system's country list which uses "India").

How they happen: Free text fields that should have been dropdowns. Spell-check autocorrections. Data migration from a system with different category names. Manual overrides.

Fix strategy: (1) Define the allowed set explicitly. (2) For obvious misspellings ("Maharastra"), map to the correct value. (3) For entirely invalid values ("Alien" in Gender), flag for review — maybe there's a new category needed.

Real story: A government welfare scheme used "Caste" as a category. When data was migrated from handwritten records, OCR and manual entry created 312 unique caste category names — many of which were duplicates of each other with spelling variations. The original scheme had 12 defined categories. Data scientists spent 3 weeks (plus domain expert review) reducing 312 to the correct 12 + an "Unverified" category.

What you do: Drop 2 unnamed columns (confirmed: they are row-number artifacts from Excel export). Drop "Referral_Code_V2" column — 91% null, no business value in a CLV model. Rename 4 inconsistently named columns to snake_case standard (CustomerID → customer_id, OrderDate → order_date).

Dataset now: 85,000 × 15 columns. Cleaner structure, consistent naming.

Day 2: Fix Data Types (4 hours)

Challenge: "order_date" column has THREE different date formats mixed together — the 3 legacy systems used different formats. You run a format detection script and find: 38,000 rows in "DD/MM/YYYY", 31,000 in "DD-MMM-YYYY", 16,000 in "YYYY-MM-DD".

What you do: Write a parsing function that handles all 3 formats and converts to a unified DateTime (ISO 8601). Test on 100 rows manually, then run on all 85,000. Verify: 0 parsing failures, no nulls introduced.

Also: Strip ₹ and commas from "order_amount", convert to Float. Flag customer_id as String (to preserve any leading zeros).

Day 3: Deduplication (2 hours)

What you find: 847 fully duplicate rows (1.0% of data) — confirmed artefacts from the 3-system merge. Additionally: 234 rows share an order_id but have different amounts (possible data corruption during merge).

Decision: Remove 847 exact duplicates immediately. Escalate the 234 conflicting order_id rows to the engineering team for source-system verification. Mark these as "Pending_Review" rather than guessing which version is correct.

Lesson learned: Never assume which conflicting duplicate is "right" without checking the source system. Guessing wrong creates worse errors than the original duplicates.

Days 4–5: Missing Value Treatment (Full Day)

"customer_age" column: 8% missing → Pattern analysis: missingness is random across all city tiers and purchase amounts. Decision: Impute with median age per city tier (Tier-1 median: 29, Tier-2: 32, Tier-3: 35). Document: "Age imputed using city-tier median; flagged in 'age_imputed' column."

"product_category" column: 3% missing → Mode imputation ("Fashion" is most common at 31% of orders). Note: This introduces a slight Fashion over-representation. Acceptable for CLV model but documented.

"delivery_date" column: 15% missing → IMPORTANT DISCOVERY: All rows with missing delivery_date have order_status = "Cancelled" or "Returned". This is NOT missing data — it's expected! Cancelled orders never get a delivery date. Decision: Fill with "N/A" as a category. Add note to ML team: "Delivery_Date absence is a feature, not missing data."

Day 6: Invalid Value Detection (6 hours)

23 orders with amount = ₹0: Verify with payment team → These are 100% discount gift orders. Completely valid! Keep them AND add a flag "is_gift_order = True". They're important for the CLV model because gifting behaviour IS a customer loyalty signal.

12 orders with amount > ₹2,00,000: Cross-reference with payment gateway records. 11 are valid bulk B2B orders from corporate accounts. 1 is a decimal error: ₹2,50,000 should be ₹2,500. Fix the 1. Keep the 11 but flag as "order_type = B2B" for separate treatment in the CLV model.

5 customers with age = 0: Cross-reference with date_of_birth column. All 5 have a valid birth date — the age field was miscalculated during a system migration. Recalculate age from DOB. Problem solved without data loss.

Day 7: Standardisation (3 hours)

City column: 47 unique text variants → 28 actual cities. Map: "mumbai", "MUMBAI", "Bombay" → "Mumbai". "bangalore", "Bengaluru", "BANGALORE" → "Bengaluru". etc. All now Title Case.

Product_Category: "fashion", "Fashion", "FASHION", "Clothing", "clothes" → "Fashion". Run similar consolidation for all categories. 71 variants → 12 standard categories.

Phone numbers: Standardise to 10-digit no-prefix format. Flag invalid phones (too short, wrong starting digit).

Day 8: Outlier Analysis (4 hours)

IQR analysis on order_amount: Upper fence = ₹14,800. 99 orders above this. Investigate: 95 are valid (confirmed B2B bulk orders, already flagged in Day 6). 4 are errors. Fix the 4.

Orders per customer analysis: 2 customers have 2,400+ orders each (vs average of 8). Investigate: These are testing accounts used by the QA team. Decision: Remove these accounts from the CLV dataset. They're not real customers.

Day 9: Final Validation + Report (2 hours)

Final dataset: 84,012 rows (98.8% of original, high retention) × 16 columns (2 dropped, 1 added: "is_gift_order").

Data Quality Report delivered: 12-page document covering every cleaning decision, why it was made, what was found, what was changed, what was escalated, and what known caveats remain. Plus: recommended additional features for the ML team to engineer from the clean data.

ML team's response: "This is the cleanest dataset we've received from this source. We can actually trust the model we'll build on this."

Reference

Master Cheatsheet

Every essential term from this guide, explained in plain English. Bookmark this page for revision.

Data
Any recorded observation: numbers, text, images, dates, locations. Raw material for AI.
Dataset
A structured table of data. Rows = individual records. Columns = attributes.
Primary Data
Collected by you, for your specific question. Fresh, relevant, but expensive.
Secondary Data
Collected by someone else. Cheaper, larger, but not designed for your question.
Structured Data
Organised in rows and columns. 20% of world data. Easy for computers.
Unstructured Data
Free-form: emails, images, audio. 80% of world data. Needs AI to process.
API
A standardised connector to access another system's data. Like a digital waiter: you send an order, it brings back data.
Data Lake
Raw, unprocessed data storage in any format. Everything in, sorted later.
Data Warehouse
Clean, structured, organised data ready for analysis. The well-sorted library.
ETL
Extract (get data), Transform (clean it), Load (put it in the warehouse). The classic data pipeline.
GIGO
Garbage In, Garbage Out. Bad data → bad model, always, regardless of algorithm quality.
NULL / NaN
A missing value. The field exists but has no value in it.
Imputation
Filling in missing values with estimated replacements.
MCAR
Missing Completely at Random. Pure chance, no pattern to missingness. Safest to impute.
MAR
Missing at Random (conditional on another variable). Use other columns to guide imputation.
MNAR
Missing NOT at Random. Missing because of the value itself. Most dangerous — can introduce bias.
Outlier
A value extremely different from the rest. Could be an error OR the most important signal.
IQR
Interquartile Range. Q3 - Q1. Distance between 25th and 75th percentile. Used to define outlier boundaries.
Z-Score
How many standard deviations a value is from the mean. |Z| > 3 → investigate as outlier.
Winsorising
Capping outliers at the fence boundary instead of removing them. Reduces influence without data loss.
Normalisation
Scaling values to 0–1 range. Min becomes 0, Max becomes 1.
Standardisation
Transforming data to mean=0, std=1 using Z-score. Different from "normalise" — common confusion!
Log Transform
Applying logarithm to compress wide-range data. ₹1Cr and ₹1000 become much closer in scale.
Schema
A blueprint: what type, format, and rules each column must follow.
Deduplication
Finding and removing or merging duplicate records.
Data Contract
Formal agreement on what valid data looks like between data producers and consumers.
Feature
A column used as INPUT for a machine learning model. Height, age, income = features.
Target / Label
The column you're trying to PREDICT. "Will this customer churn?" — churn = the target.
EDA
Exploratory Data Analysis. Understanding your data through charts and statistics before modelling.
Regex
Regular Expression. A pattern that defines what valid text looks like. Used to validate formats.
Forward Fill
In time-series: fill a gap with the previous observed value.
KNN Imputation
Fill missing values using averages from the K most similar records.
MICE
Multiple Imputation by Chained Equations. Statistical method for principled missing value filling.
One-Hot Encoding
Converting a category column into multiple binary (0/1) columns — one per category. Required for most ML models.
Feature Engineering
Creating new columns from existing ones to give models more useful information.
Data Pipeline
Automated sequence: collect → clean → transform → load → model → predict.
Data Profiling
Automated statistical summary of a dataset — the first thing you always do.
Cross-Field Validation
Checking consistency between two or more columns together (e.g., DOB vs Employment Start Date).
